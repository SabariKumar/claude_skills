# document-code

Writes and audits Python docstrings and per-directory README files.

---

## Step 0: Ask the user what to document

Before reading any files or doing any work, always ask the user which of the following they want. Present these as distinct options to choose from:

1. **Docstrings only** — add/fix docstrings in the source files; skip READMEs and diagrams.
2. **README only** — write/audit the `README.md` for the directory; skip docstrings and diagrams.
3. **Both** — docstrings and README; skip diagrams.
4. **README + diagram** — write/audit the README and generate a scientific SVG diagram from it.
5. **Everything** — docstrings, README, and diagram.

Diagrams are **not** generated by default. Only generate a diagram if the user explicitly selects an option that includes it.

If the user's request already makes the scope unambiguous (e.g. "add docstrings to this file" or "write a README for this module"), skip the prompt and proceed with the implied scope — but still do not generate a diagram unless they asked for one.

---

## Docstrings

All functions and methods must use this exact format:

```python
def example(param_a: str, param_b: int) -> None:
    """
    One-line summary of what the function does.

    Params:
        param_a: str : description
        param_b: int : description
    Returns:
        description of return value, or None
    """
```

**Rules:**
- Always include `Params:` and `Returns:` sections — even if `Params: None` or `Returns: None`.
- Type annotations belong in the signature; repeat the type in the docstring after the colon for readability.
- The one-line summary must be concise and imperative: "Compute...", "Return...", "Load...", "Build...".
- Do not write multi-sentence summaries in the one-liner. If more context is needed, add a blank line after the summary and a prose paragraph before `Params:`.

**Edge cases:**
- `@property` methods: include `Returns:` but `Params: None`.
- `__init__`: document in the class-level docstring, not `__init__` itself. Class docstring describes the object; use `Params:` for constructor args.
- `@classmethod` / `@staticmethod`: same format; omit `cls`/`self` from `Params:`.
- Overriding a parent method with no behavioral change: a one-liner referencing the parent is acceptable; full `Params:`/`Returns:` still required.

## READMEs

Every subdirectory containing code must have a `README.md`.

**A reader unfamiliar with the codebase should be able to understand the module without opening any source files.**

### Required sections (in order)

1. **Purpose** — What problem this module solves and where it fits in the overall pipeline. One paragraph.
2. **Module contents** — One paragraph per file. Cover: what it does, key classes/functions, and non-obvious design decisions.
3. **Data contracts** — The shape, dtype, and meaning of inputs and outputs for the main entry points. Use inline code or a small table.
4. **Critical parameters or constraints** — Anything that silently breaks correctness if misconfigured (e.g., a `max_neighbors` cap that truncates graphs without warning). Flag these explicitly.
5. **Dependencies on other modules** — What this module consumes from siblings and what it produces for downstream modules.

### Tone and depth

- Write for a reader who knows Python and ML but has never seen this codebase.
- Do **not** summarise what code does line-by-line. Explain *why* the code is structured the way it is.
- When a design choice looks surprising (e.g., using a heterograph instead of a homogeneous graph), give one sentence of motivation.
- Keep prose tight. A 200-line module warrants roughly 300–500 words of README. A 2000-line module might warrant 800–1200.

## Workflow: auditing existing documentation

When asked to audit rather than write from scratch:

1. Read all source files in scope.
2. Identify functions/methods missing docstrings entirely.
3. Identify docstrings that are missing `Params:` or `Returns:` sections, have mismatched parameter names, or have non-imperative summaries.
4. Identify directories missing a `README.md`.
5. Identify READMEs that are missing required sections or describe *what* code does rather than *why* it is structured that way.
6. Report findings grouped by file, then fix them or propose fixes depending on the user's request.

---

## Workflow: writing documentation for new code

When asked to document a function, file, or directory:

1. Read the source in full before writing anything.
2. For each function/method: infer intent from implementation; write the docstring using the format above.
3. For each directory: identify the module's role in the broader pipeline, then write the README covering all five required sections.
4. Do not invent behavior — if a function's purpose is ambiguous, note the ambiguity and ask before writing.

## Workflow: generating scientific diagrams

Only run this workflow when the user selected an option that includes a diagram (options 4 or 5 from Step 0).

**Before writing any SVG code, read the `svg-scientific-style` skill** in full. It contains the canonical marker definitions, color palette, typography, and shape specs that all figures must follow.

### Figure style by location

| README location | Figure style |
|---|---|
| Repo root (`README.md`) | **Overview** — high-level data flow for a computational chemistry audience; no hyperparameters, no layer counts. Show the pipeline at a conceptual level. |
| Subdirectory (`<subdir>/README.md`) | **Detailed** — architecture reference with layer counts, tensor dimensions, key parameters, and training config, for a reader working on the code. |

### Steps

1. Read the README in full (write it first if it doesn't exist yet).
2. Identify the one diagram that best communicates the module's structure or pipeline. Do not generate multiple diagrams per README.
3. Select the correct style (overview vs. detailed) based on the README's location.
4. Read `/mnt/skills/user/svg-scientific-style/SKILL.md` to load all style rules.
5. Write the SVG following those rules exactly — palette, markers, typography, canvas dimensions.
6. Save the SVG file **in the same directory as the README**, named `diagram.svg`.
   - Root: `diagram.svg` alongside `README.md`
   - Subdirectory: `<subdir>/diagram.svg` alongside `<subdir>/README.md`
7. Do not embed the SVG inline in the README. Just save the file.
