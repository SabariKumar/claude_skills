# claude-skills

A personal library of reusable Claude Code skills. Each skill is a SKILL.md file that Claude Code loads on demand via the `Skill` tool, giving any Claude Code session access to specialized behavior without cluttering the global CLAUDE.md.

---

## Purpose

This repo collects skills that are too domain-specific or too detailed to live in a global config, but useful enough across multiple projects to warrant a shared home. The skills here cover three areas: writing quality (AI-ism removal), code documentation (docstrings + READMEs), and scientific figure generation (SVG style). A fourth file captures project-level Claude instructions carried over from the `peptide-electrostatics` repo.

Skills are not code — they are structured markdown documents with a YAML frontmatter header that the Claude Code skill system reads to decide when to trigger a skill automatically and what it does when invoked.

---

## File contents

**`avoid-ai-writing.md`** — Instructs Claude to audit and rewrite text to remove AI writing patterns ("AI-isms"). Operates in two modes: `rewrite` (default, fixes and returns clean copy) and `detect` (flags only, no changes). The skill contains a three-tier vocabulary list adapted from the `brandonwise/humanizer` project, a context-profile tolerance matrix (six profiles: `linkedin`, `blog`, `technical-blog`, `investor-email`, `docs`, `casual`), and a severity taxonomy (P0 credibility killers → P2 stylistic polish). The design choice to separate vocabulary tiers from structural rules matters: swapping flagged words without fixing sentence rhythm and paragraph uniformity leaves text that still reads as AI-generated, so the skill explicitly covers both.

**`document-code.md`** — Instructs Claude to write or audit Python docstrings and per-directory README files. Supports five scopes selectable at invocation time: docstrings only, README only, both, README + SVG diagram, or everything. Docstring format is strictly specified (imperative one-liner, `Params:` and `Returns:` sections always present, types repeated in the docstring body). README format follows five required sections in order: Purpose, Module contents, Data contracts, Critical parameters, Dependencies on other modules. The skill conditionally triggers the `svg-scientific-style` skill when diagram generation is requested — it does not embed the SVG inline, only saves it alongside the README.

**`svg-scientific-style.md`** — Style reference for generating SVG scientific figures that match an established aesthetic (Inkscape 1.4, 600 DPI, A4 canvas in millimeter coordinate space). Covers two detail levels: `overview` (conceptual pipeline for a computational chemistry audience, no hyperparameters) and `detailed` (architecture reference with layer counts, dimensions, and training config). Contains verbatim `<defs>` blocks for all four canonical marker types (Triangle, Triangle-sm, ArrowTriangleStylized, ColoredDot), a full color palette table, typography specs, shape recipes, and layout arithmetic for spine alignment and lane spacing. The palette and markers are fixed across all figures to ensure visual consistency — figures generated by Claude should look hand-made in Inkscape, not algorithmically generated.

**`repo_claude.md`** — Project-level CLAUDE.md content from the `peptide-electrostatics` repo. Captures: the `pixi` dependency management convention, the docstring format (same as `document-code.md`), branch and issue hygiene protocol (new branch before major features, GitHub issue summary before code, PR summary template on completion), and the README content standard. Not a skill in the Claude Code skill sense — it is raw CLAUDE.md content meant to be copied into a project's `.claude/CLAUDE.md` or `CLAUDE.md`.

## Skill file format

Each skill file uses this structure:

```
---
name: <skill-name>
description: >
  One or more sentences describing when to trigger the skill.
  This text is used by the Claude Code skill system for matching.
---

# <Skill Title>

Markdown body with the full skill instructions...
```

The `description` field drives automatic triggering — it should list representative user phrases and task types. The body is loaded in full when the skill is invoked, so it can be as long as needed. There is no length limit, but skills that are too broad tend to conflict with each other.

## Critical constraints

- **Diagram generation is not the default** in `document-code.md`. It only runs when the user selects an option that explicitly includes a diagram (options 4 or 5). Adding diagram generation to option 1, 2, or 3 would cause it to run on every documentation request, which is expensive and usually unwanted.

- **The `svg-scientific-style.md` skill must be read before writing any SVG code** when called from `document-code.md`. The style rules — especially the canonical `<defs>` block and the spine alignment arithmetic — are not in Claude's training data for this specific figure aesthetic. Skipping the read produces figures that look inconsistent with hand-edited Inkscape files.

- **Skill frontmatter `name` must match the invocation name** used in Claude Code. If a skill file is renamed, update the `name` field in the frontmatter to match, or the `Skill` tool will fail to find it.

- **`repo_claude.md` is not a skill** — it has no frontmatter and cannot be invoked via the `Skill` tool. It is a raw CLAUDE.md file kept here for reuse across projects.

## Installation

Claude Code loads user skills from `~/.claude/skills/`. Each skill must be a subdirectory named after the skill, containing a file named `SKILL.md`. The files in this repo use descriptive filenames for version control; the install step renames them.

### Mac (local machine)

```bash
# Clone or pull the repo
git clone <repo-url> ~/Documents/Projects/claude_skills
cd ~/Documents/Projects/claude_skills

# Create the skills directory if it doesn't exist
mkdir -p ~/.claude/skills

# Install each skill as a subdirectory
for skill in avoid-ai-writing document-code svg-scientific-style; do
  mkdir -p ~/.claude/skills/$skill
  cp ${skill}.md ~/.claude/skills/$skill/SKILL.md
done
```

Restart Claude Code (or reload the window in VS Code) after installing. Skills take effect immediately in new sessions.

### Linux remote machine (VS Code Remote Development)

The VS Code Remote Development extension runs the Claude Code extension process on the remote host, so skills must be installed on the remote machine's filesystem — not your local Mac.

**Option A — clone the repo directly on the remote machine** (recommended if you have internet access there):

```bash
# On the remote machine
git clone <repo-url> ~/claude_skills
cd ~/claude_skills

mkdir -p ~/.claude/skills
for skill in avoid-ai-writing document-code svg-scientific-style; do
  mkdir -p ~/.claude/skills/$skill
  cp ${skill}.md ~/.claude/skills/$skill/SKILL.md
done
```

**Option B — transfer from your Mac with `rsync`**:

```bash
# Run this on your Mac
# First, install locally so you can rsync the structured layout
for skill in avoid-ai-writing document-code svg-scientific-style; do
  mkdir -p ~/.claude/skills/$skill
  cp ${skill}.md ~/.claude/skills/$skill/SKILL.md
done

# Then push the skills directory to the remote machine
rsync -av ~/.claude/skills/ <user>@<remote-host>:~/.claude/skills/
```

### Keeping skills up to date

When skill files change in this repo, re-run the copy loop for the skills that changed, then reload the VS Code window (`Cmd+Shift+P` → `Developer: Reload Window`) so Claude Code picks up the new content.

## Dependencies

These skills are self-contained markdown documents with no runtime dependencies. The `document-code.md` skill cross-references `svg-scientific-style.md` at generation time (it instructs Claude to read that file before writing SVG code), so both files must be present in the same skill library for diagram generation to work correctly.

No build step, no package manager, no install required. Skills are consumed directly by Claude Code.
